# Knowledge Vault Candidate Preview 4J-3B / 4J-3C

## A. Purpose

4J-3B / 4J-3C adds a preview-only Knowledge Vault candidate layer to the parsed-DOCX Source Library workflow.

The goal is to make the workflow feel closer to:

```text
Input -> Automatic Classification -> Keep Records by Tag -> Textbook Request -> DOCX Output
```

This sprint shows how parsed-DOCX classification results could become organized tag records or Knowledge Vault entries later. It does not save records or create final KnowledgeCards.

## B. Relationship To 4J-3A

4J-3A added deterministic parsed-DOCX `Classification & Tag Preview`.

4J-3B / 4J-3C consumes that preview output and maps it into `Knowledge Vault Candidate Preview` records. The new mapper depends on classification preview state and does not inspect external providers or call persistence.

## C. Candidate Mapping Rules

The preview is generated by `ParsedDocxKnowledgeVaultCandidateMapper`.

Inputs are existing frontend state only:

- parsed-DOCX availability
- classification preview status
- suggested marketing tags
- source type hint
- textbook relevance hints
- source signals from classification preview

Candidate records include:

- candidate ID
- tag label
- tag group/category
- source relationship
- suggested vault use
- confidence
- reason
- linked signals
- review required
- persistence status: `preview_only`

The mapper blocks when:

- no parsed DOCX exists
- classification preview is missing or not ready
- tag signals are insufficient

## D. Suggested Vault Use Behavior

Suggested vault uses are conservative local hints:

- service quality terms can become `concept_record` and `textbook_section_input`
- customer experience terms can become `concept_record` or `evidence_record`
- satisfaction and loyalty terms can become `concept_record`
- case, company, or example signals can become `teaching_case_record`
- method, research, or evidence signals can become `evidence_record`

Low-confidence tags remain review-only and are not promoted into saved records.

## E. What Changed

- Added a pure TypeScript Knowledge Vault candidate preview mapper.
- Added a compact `Knowledge Vault Candidate Preview` panel below the classification preview.
- Added a guided action path step for `Preview Knowledge Vault Candidates`.
- Added preview-only, not-saved, no-AI, no-citation-finality, and human-review-required notices.
- Added mapper-level and Source Library QA assertions.

## F. What Did Not Change

This sprint did not change:

- backend commands
- schema
- migrations
- persistence behavior
- DOCX parser behavior
- DOCX export behavior
- metadata mutation
- metadata apply expansion
- compact SourceCard apply
- SourceCard title/authors/year/sourceType mutation through correction workflows
- `citationText`
- APA-final verification
- live Crossref/OpenAlex/DOI/ISBN calls
- API keys
- AI/API behavior
- PDF/OCR implementation
- dependencies
- existing save/review gates

No Knowledge Vault candidate is saved automatically.

## G. Guardrails

Visible guardrails include:

- preview only
- no automatic save
- human review required
- no AI used
- SourceCard `citationText` not overwritten
- APA-final not supported
- DraftArtifact remains mock/not-final
- DOCX export remains gated
- no citation finality

## I. Remaining Limitations

- Candidate records are local deterministic previews only.
- Suggested vault use is a routing hint, not a saved Knowledge Vault decision.
- No reviewed classification state is persisted.
- No final KnowledgeCards or textbook prose are generated.
- Confidence remains a local UI hint and should not be treated as academic certainty.

## J. Recommended Next Sprint

Recommended next sprint: 4J-3D local review state for vault candidates.

Focus on allowing users to approve, reject, or defer Knowledge Vault candidate records locally before exposing existing explicit save paths. Keep save behavior explicit and do not mutate SourceCard metadata.
